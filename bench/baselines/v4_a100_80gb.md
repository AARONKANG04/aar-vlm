# `v4` shared-memory padding for bank conflicts on A100 80GB

GPU: A100 80GB PCIe, 108 SMs, sm_80.
v4 = a copy of v3's kernel with shared memory inner dims padded by 4 floats: `AsT[BK][BM+4]` and `Bs[BK][BN+4]`. Code in [cppsrc/ops/matmul.cu](../../cppsrc/ops/matmul.cu). v3 stays around as the no-padding baseline.

## Why

After v3, sq8192 sat at 89% of `cublas_sgemm`. The v3 writeup pointed at shared-memory bank conflicts as one likely contributor to the remaining ~11pt. v4 tests that hypothesis with the canonical "+4 pad" trick and tries to measure the result.

## Conflict analysis going in

Before changing anything, I worked through the access patterns to predict what padding would and wouldn't fix.

For BM=BN=128, BK=16, TM=TN=8, with 256 threads per block:

| operation | conflict | reason |
|---|---|---|
| AsT STS (4 scalar STS per thread, transposed during store) | 4-way | k_local ∈ {0,4,8,12} all hit same bank because 4·BM = 512 ≡ 0 mod 32 |
| AsT LDS (2 float4 LDS per K-iter) | none | the ty=0/1 broadcast groups land on different banks |
| Bs STS (1 float4 STS per thread) | 4-way | tx*4 mod 32 has period 8 |
| Bs LDS (2 float4 LDS per K-iter) | 4-way | tx*8 mod 32 has period 4 |

Padding the inner dimension shifts the across-row stride from a multiple of 32 to non-multiple. That helps when the conflict is across rows. But within a single LDS instruction every thread in the warp uses the same kk, so within-row conflicts are invisible to row-stride changes.

Walking through what padding can touch:

| operation | does +4 padding help? |
|---|---|
| AsT STS | yes, partially. k_local·(BM+4)·4 mod 32 spreads from {0} to {0, 16}. 4-way → 2-way. |
| AsT LDS | no change (it was already conflict-free). |
| Bs STS | no. Conflict is within-row; padding row width is invisible. |
| Bs LDS | no. Same reason. |

So I went in expecting modest gains, mostly from cutting the AsT STS conflict in half. The Bs LDS conflict (which I think is the dominant LDS cost) wouldn't move.

## What changed

`matmul_tiled_kernel_v4` is structurally identical to v3 with two differences:
1. SMEM declarations: `__shared__ float AsT[BK][BM+4]` and `__shared__ float Bs[BK][BN+4]`
2. The `mma_compute_v3` helper takes references to fixed-shape arrays (`const float (&)[BK][BM]`), which won't accept the padded type. So the v4 kernel inlines the mma compute directly. Same logic, just with the padded stride flowing through the compiler.

Everything else — float4 GMEM loads, transpose-on-store for AsT, no bounds checks, dispatcher heuristic, alignment guards, fallback to `matmul_v2_launch` — is the same as v3.

I tried to verify the conflict count drop with `ncu`, but the pod blocks GPU performance counters (`ERR_NVGPUCTRPERM`). So we have timing data only. That's annoying because it means we can't directly say "the AsT STS conflicts went from 4-way to 2-way" — we can only compare end-to-end times against v3 and infer.

A consistent +1 to +3pt across every v3-fast-path shape. Lower end of the predicted 2-5pt range. No regressions on the fallback shapes (they hit the same v2 kernel as before).

Four shapes now sit at 90% of `cublas_sgemm` or above: sq8192 (90.9), ffn_down (91.0), tall (90.3), wide (90.5). sq8192 in absolute terms is 63.7 ms vs cuBLAS's 57.9 ms, so within ~9pt of the cuBLAS asymptote.

One thing I noticed: the 64-tile shapes (sq1024, ffn_down, tall, wide) gained about 2.6-2.9pt while the 128-tile shapes only gained 1.0-1.9pt. I think this is because each 64-tile block has less compute per K-iter, so the AsT STS overhead is a bigger fraction of total time. The conflict reduction hits harder when the conflict's contribution to runtime is bigger. Not 100% sure but the pattern fits.

## What didn't move and why

The remaining ~9pt gap to cublas_sgemm on sq8192 still comes mostly from things v4 didn't tackle:

- **Bs LDS 4-way conflict.** Padding can't fix this — the conflict is within a single warp instruction at fixed kk, so it's invariant to row stride. Fixing it would need either a thread-id-aware swizzle (CUTLASS-style) or a different tiling structure where the access pattern doesn't have the period-4 issue. Not easy.
- **Single-buffered shared memory.** Every K-tile still does load → sync → compute → sync. Two SMEM slots + ping-pong (v5) lets the next load run in parallel with the current compute.
- **Loads going through registers.** On Ampere, `cp.async` (v6) bypasses the register file, freeing it for compute.

The 32-tile shapes can't benefit from v4 at all because they fall back to v2's kernel (TM=2 fails the float4 register fill alignment).

## Some things worth flagging

- I couldn't directly verify the conflict count change because the pod blocks `ncu`. The timing data is the only proof, and the 2-3pt gain is consistent with the predicted partial fix, but I can't actually point at "AsT STS went from 4-way to 2-way" with measured numbers. If I get on a box where ncu works, that'd be worth doing in retrospect.
- The +4 amount was chosen because it preserves float4 alignment (it's a multiple of 4) while breaking the 32-bank period (132 mod 32 = 4). Other multiples-of-4 padding (e.g., +12, +20) would behave the same for AsT STS — they all keep the same residual 2-way conflict. The "real" fix would need a non-multiple-of-4 pad, but that breaks float4 LDS alignment, so it's not viable here.
- v3 stays as the no-padding baseline. If a later kernel introduces a regression around bank conflicts, having v3 vs v4 side by side on the bench should make it visible.
- Sanity check on the headroom: `cublas_sgemm` at sq8192 hits 18.99 TFLOPS (97% of A100's 19.5 TFLOPS FP32 peak). v4 hits 17.26 TFLOPS = 88.5% of peak. The runway left is small enough that v5 + v6 are probably the last meaningful FP32 levers before tensor cores become the only remaining option.
