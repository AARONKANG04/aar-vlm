# `v3` float4 vectorized loads + transposed shared layout on A100 80GB

GPU: A100 80GB PCIe, 108 SMs, sm_80.
v3 = a new kernel `matmul_tiled_kernel_v3<BM,BN,BK,TM,TN>` that lives next to v1/v2's `matmul_tiled_kernel`. Code in [cppsrc/ops/matmul.cu](../../cppsrc/ops/matmul.cu). v1/v2 are kept around as regression baselines.

## Why

After v2 fixed the dispatch problem, the shapes that picked the 128-tile (sq2048+, qkv_proj, ffn_up, big_ffn) flatlined around 60-72% of cublas_sgemm. The dispatcher couldn't help further. The gap was inside the kernel body itself.

Three things looked obviously fixable in the inner loop:

1. Scalar global loads. v1/v2 issue 8 separate scalar loads per thread per K-tile, on each side. cuBLAS issues 16-byte vector loads (`LDG.E.128`), which is half the instruction count for the same bandwidth.
2. Strided register fills for A. The `As[BM][BK]` layout means `a_reg[i] = As[ty*TM + i][kk]` strides by `BK` floats along `i`, which kills any chance of doing a vector LDS into registers.
3. Bounds checks every K-iter. Even on shapes where every tile is exactly aligned, the predicates still get evaluated inside the hot loop.

v3 fixes all three.

## What changed

### Float4 global loads

Each thread now loads 2 `float4`s per K-tile instead of 8 scalars, on each side. With BM=128, BK=16, THREADS=256: BM\*BK = 2048 floats = 512 float4s, divided among 256 threads = 2 float4 per thread. Same math for B.

I wasn't sure the compiler would actually emit the 16-byte loads until I checked. After building, `cuobjdump --dump-sass build/bench_matmul | grep -c LDG.E.128` returned 6, which matches what I expected: 4 from the 128-tile kernel (2 unrolled iterations for A and 2 for B) plus 2 from the 64-tile kernel. If those had stayed as `LDG.E.32` the whole change would have been pointless.

### Transposed shared memory for A

The bigger structural change: store A as `AsT[BK][BM]` instead of `As[BM][BK]`. The register fill becomes `a_reg[i] = AsT[kk][ty*TM + i]`, which is contiguous along `i`. That admits a `float4` LDS per 4 register elements.

For TM=8 we go from 16 scalar shared-memory loads per K-iteration (8 per side in v1/v2) to 4 float4 shared-memory loads (2 per side). Quarter the LDS instruction count for the same throughput.

The catch: the GMEM read pattern produces 4 contiguous K-values for one M-row, but storing that into `AsT` (transposed) needs 4 scattered scalar STS rather than a single float4 STS. The transpose has to happen somewhere. Doing it on the store side (a one-time scalar fan-out) preserves the contiguous LDS on the hot path. STS bandwidth is high enough on A100 that this isn't a bottleneck in practice.

### No bounds checks

v3 only runs when the shape is fully aligned: M%BM==0, N%BN==0, K%BK==0, plus K%4==0 and N%4==0 for the float4 alignment. If any of those fail, `matmul_v3_launch` calls `matmul_v2_launch` as a fallback. Of the 14 bench shapes, only `vocab_head` (N=65) hits that fallback because of the alignment.

### Tile size constraint

The float4 LDS for register fills needs TM%4==0 and TN%4==0. The 32-tile (TM=TN=2) doesn't satisfy that, so v3 also falls back to v2 for the 32-tile shapes (sq256, sq512, tiny_M). I could write a third kernel with scalar register fills for 32-tile, but those shapes are already at decent percentages (sq256 actually beats cuBLAS) so I don't think it's worth the extra code.

The biggest jumps are on the large 128-tile shapes that were stuck at v1's compute ceiling. sq8192 went from 72% to 89% of cublas_sgemm. The asymptote moved by about 16 percentage points.

In absolute numbers, sq8192 v3 takes 64.93 ms vs cuBLAS's 57.57 ms, so within 11% of the most-tuned SGEMM I have to compare against.

The 64-tile shapes (ffn_down, tall, wide) got smaller but real bumps of around 7pt each. The 128-tile transformer shapes (qkv_proj, ffn_up, big_ffn) all jumped 14-16pt.

The 32-tile shapes didn't move because v3 is bypassed for them. Their numbers come straight from v2's fallback.

## What didn't move and why

The remaining ~11pt gap on sq8192 (and similar gaps elsewhere) probably comes from things v3 didn't tackle:

- Shared-memory bank conflicts. The LDS reads of `AsT` and `Bs` still hit the same 2-way and 4-way bank conflict patterns I noted in the v1 writeup. Padding the inner dims (`AsT[BK][BM+4]`, `Bs[BK][BN+4]`) should fix it. Small change, planned for v4.
- Single-buffered shared memory. Every K-tile syncs twice: load, sync, compute, sync. Two shared-memory slots and ping-pong between them lets the next load run alongside the current compute. Planned for v5.
- Loads still go through the register file. On Ampere, `cp.async` copies GMEM directly into SMEM, bypassing registers. Planned for v6.

Combined, those should close most of what's left of the gap to cublas_sgemm.

The 32-tile shapes are stuck for different reasons:
- sq256 already beats cublas_sgemm because at this size cuBLAS's per-call overhead dominates.
- sq512 with 32-tile gives 256 blocks (good occupancy) but each block's compute is short. Float4 wouldn't help even with TM=2 → scalar register fills.
- vocab_head: N=65 is non-aligned, so it falls to v2 forever unless I write a non-square tile variant.
- tiny_M: M=128 only generates 4 row tiles regardless of N. cuBLAS likely uses split-K parallel reduction here. That's a different kernel entirely.

## Some things worth flagging

- I should have checked the SASS earlier. The `cuobjdump | grep LDG.E.128` step is fast and catches the case where the compiler silently scalarizes everything. Going forward I want to make this a default check on any new kernel.
- The fallback to v2 has to call `matmul_v2_launch`, not duplicate the dispatcher logic. v2 already has the tile-size heuristic. v3 just pre-checks alignment and either runs its own fast path or hands off. Keeping the dispatch heuristic single-sourced makes it easier to keep adding versions on top.
- Sanity check on the headroom: cuBLAS at sq8192 hits 19.10 TFLOPS (98% of A100's 19.5 TFLOPS FP32 peak). v3 hits 16.93 TFLOPS = 87% of peak. So I'm losing 11pt to cuBLAS but only 13pt to the hardware ceiling. There isn't a huge runway left on the FP32 path before tensor cores become the only remaining lever.
