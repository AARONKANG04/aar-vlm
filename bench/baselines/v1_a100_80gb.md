# `v1` kernel matmul on A100 80GB

GPU: A100 80GB PCIe, 108 SMs, sm_80. FP32 peak around 19.5 TFLOPS, TF32 peak around 156 TFLOPS.
Harness: `bench_matmul`, 50 iters with 10 warmup, `cudaEvent` timing. The pod blocked clock locking, so p99 can be a bit noisy.
v1 = `matmul_tiled_kernel` in [cppsrc/ops/matmul.cu](../../cppsrc/ops/matmul.cu). 128×128 block, 8×8 register tile, BK=16, scalar global loads, no prefetch.

## Notes

cublas_sgemm hits 19.18 TFLOPS at sq8192, which is about 98% of A100's FP32 peak. So that's basically the ceiling.

v1 lands at 71.9% of cublas_sgemm at sq8192 (13.80 TFLOPS), and seems to plateau near 70% on the large square shapes. The remaining gap is probably a mix of shared-memory bank conflicts, scalar global loads instead of float4, no double-buffering, and per-load bounds checks running inside the hot K-loop. I want to verify with `ncu` before fixing.

Small shapes (sq256, sq512) do badly because the 128×128 block size leaves most SMs idle. sq256 only launches 4 thread blocks for 108 SMs.

Skinny shapes (vocab_head, tall, wide) lose to tile padding. vocab_head has N=65 but the kernel still allocates a full BN=128 column tile, so roughly half the column work is wasted.

The %tf32 column is reference only. cuBLAS uses tensor cores there and v1 is plain FP32 FMAs, so the comparison isn't fair until I write a tensor-core kernel.
