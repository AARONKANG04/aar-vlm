# `v2` size-adaptive tile dispatch on A100 80GB

GPU: A100 80GB PCIe, 108 SMs, sm_80.
v2 = the same `matmul_tiled_kernel` from v1, but templated on `<BM, BN, BK, TM, TN>` with a runtime dispatcher that picks a tile size per call. Code in [cppsrc/ops/matmul.cu](../../cppsrc/ops/matmul.cu). v1 stays around as the 128-tile instantiation so I can keep using it as a regression baseline.

## Why bother

Looking at the v1 numbers, two patterns jumped out. Small problems like sq256 only launched 4 thread blocks for 108 SMs, so most of the GPU sat idle. And skinny problems like vocab_head (N=65) padded each column tile up to 128, so roughly half the per-block work was wasted multiplying past the real data.

cuBLAS handles both of these by picking tile sizes per shape. So the smallest version of "do what cuBLAS does" is: have a few tile sizes available, dispatch at runtime.

## What I tried

### Attempt 1: pick by min(M, N)

```
min(M,N) <= 128 -> 32x32
min(M,N) <= 384 -> 64x64
else            -> 128x128
```

Worked great for skinny shapes. vocab_head went 47% → 65%, tall went 49% → 80%, ffn_down 67% → 80%. But sq512 and sq1024 didn't move because their min is 512 and 1024, so they fell into the 128-tile branch. They only launch 16 and 64 blocks respectively, which still doesn't fill 108 SMs.

### Attempt 2: pick by grid block count

```
blocks(128) >= 108 -> 128x128
blocks(64)  >= 108 -> 64x64
else               -> 32x32
```

The thinking was: pick the largest tile that still produces enough blocks to fill one SM wave. This nailed the small squares — sq512 jumped from 24% to 74%, sq1024 from 50% to 69%, and sq256 actually overtook cuBLAS at 112%. But every skinny shape regressed back to v1 levels. tall went from 80% to 49%, vocab_head 65% to 46%.

What I missed: skinny shapes hit "≥ 108 blocks" by counting alone, but each 128-tile block on a skinny problem has only 2 or 3 column tiles to chew through. With low K each block finishes fast, so the tail wave imbalance dominates. Going to 64-tile gives you ~4× more blocks, and even though arithmetic intensity per block is lower, the better SM utilization wins.

### Attempt 3: hybrid

```
small = min(M, N)
if small <= 128                                     -> 32
else if small <= 384 and blocks(64) >= SM_TARGET    -> 64
else if blocks(128) >= 2 * SM_TARGET                -> 128
else if blocks(64) >= SM_TARGET                     -> 64
else                                                -> 32
```

This is just the union of the two earlier rules in the right order. Skinny shapes (small ≤ 384) prefer the 64-tile if it gives enough blocks. Otherwise we fall to grid-aware, but with the 128-tile threshold bumped to 2*SM_TARGET so we want at least two full waves before committing. SM_TARGET is hardcoded to 108 for A100.

I had to walk through all 14 shapes by hand to convince myself the precedence was right. It is, but I wouldn't be surprised if a fourth tile size or a couple more checks would help.

## What didn't move

The shapes that picked the 128-tile (sq2048-sq8192, qkv_proj, ffn_up, big_ffn) are still in the 60-72% range. The dispatcher can't help these. They're compute-bound on a tile that's already the right size, so the gap is inside the kernel body. v3 will go after that with float4 vectorized loads, a transposed shared-memory layout for vector register fills, and bank-conflict fixes. I'm hoping to push the large-shape asymptote from ~72% toward ~85%, but I want to verify with `ncu` first.

tiny_M (49%) and vocab_head (66%) are also still stuck. tiny_M has M=128 so the grid is at most 4 row blocks regardless of N — the kernel is compute-poor and cuBLAS probably uses split-K reduction here. vocab_head has N=65 which doesn't divide cleanly into any of my tile widths, so 30-50% padding is unavoidable with square tiles. Both probably need their own dedicated kernels.

## Some things worth flagging

- SM_TARGET = 108 is hardcoded for A100. On other GPUs you'd query `cudaDeviceProp::multiProcessorCount` at runtime. Leaving it hardcoded for now since this project only targets one GPU at a time.
- I kept thread count at 256 for all three tile sizes (TX × TY = 16 × 16 always). That keeps the register and occupancy budget roughly constant, which made it easier to reason about why one tile beat another.
- v1 is preserved as the 128-tile instantiation of the same template, so it's byte-identical to its previous form. Useful for catching regressions while I keep changing v2 (and later v3).
